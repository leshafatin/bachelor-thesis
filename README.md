# whatever-rendering

Исследовательский стенд для сравнения стратегий рендеринга `CSR`, `SSR` и `SSG` на одном и том же наборе пользовательских сценариев. Проект показывает не только разницу в первой отрисовке, но и помогает собирать метрики производительности, события пользовательского поведения и результаты A/B-распределения по стратегиям.

## Что умеет проект

- отдает страницы через `CSR`, `SSR` и `SSG`;
- собирает серверные и клиентские метрики: `TTFB`, `serverRenderMs`, `LCP`, `CLS`, `INP`;
- пишет просмотры страниц и продуктовые события в `SQLite`;
- поддерживает A/B-распределение пользователей по стратегиям через cookie;
- показывает результаты в админке и позволяет экспортировать данные в `CSV`.

## Сценарии

В репозитории есть три демонстрационных сценария:

- `hello` - статическая контентная страница;
- `report` - динамический операционный отчет с меняющимися данными;
- `catalog` - интерактивный каталог с фильтрацией и продуктовой воронкой.

Все сценарии открываются по адресу `http://localhost:3000/page/:slug`.

## Архитектура

```text
.
├── client/   # Vite + React клиент, гидратация, web-vitals, отправка метрик
├── server/   # Express сервер, стратегии рендеринга, админка, SQLite, API
├── shared/   # общие React-компоненты и типы страниц
├── deploy/   # nginx-конфиг для контейнерного запуска
└── Dockerfile
```

Ключевые части:

- `server/src/index.ts` - HTTP-сервер, API, метрики, маршруты `/page/:slug`, `/admin`, `/healthz`;
- `server/src/strategy/` - выбор и выполнение стратегии `csr | ssr | ssg`;
- `server/src/experiments/` - конфигурация и рантайм A/B-экспериментов;
- `server/src/admin/` - HTML-админка, агрегация аналитики и экспорт;
- `server/src/metrics/` - инициализация `SQLite` и работа с метриками;
- `client/src/metrics.ts` - отправка `web-vitals` на сервер.

## Как это работает

1. Пользователь открывает `/page/:slug`.
2. Сервер выбирает стратегию рендеринга: либо базовую для маршрута, либо назначенную экспериментом.
3. Сервер логирует `page_view`, считает `TTFB` и `serverRenderMs`.
4. Клиент гидратирует или монтирует React-приложение и отправляет `LCP`, `CLS`, `INP`.
5. Админка агрегирует результаты по стратегии и группе эксперимента.

## Требования

- Node.js 20+
- npm 10+

## Локальный запуск

Установить зависимости:

```bash
npm install
```

Запустить клиент и сервер в режиме разработки:

```bash
npm run dev
```

После запуска будут доступны:

- приложение и админка: `http://localhost:3000`
- Vite dev server: `http://localhost:5173`

Полезные URL:

- `http://localhost:3000/admin`
- `http://localhost:3000/page/hello`
- `http://localhost:3000/page/report`
- `http://localhost:3000/page/catalog`
- `http://localhost:3000/healthz`

## Production build

Собрать клиент и сервер:

```bash
npm run build
```

Сгенерировать статические HTML-файлы для `SSG`:

```bash
npm run ssg
```

Запустить production-сервер:

```bash
npm start
```

## Docker

Собрать и поднять контейнеры приложения и `nginx`:

```bash
docker compose up --build
```

По умолчанию:

- приложение слушает `3000` внутри контейнера;
- `nginx` публикуется на `http://localhost:80`;
- база хранится в volume `render_lab_data`.

## Переменные окружения

Сервер использует следующие переменные:

- `PORT` - порт Express-сервера, по умолчанию `3000`;
- `DB_PATH` - путь к `SQLite`-файлу, по умолчанию `server/db.sqlite` при локальном запуске;
- `NODE_ENV` - влияет на подключение клиентских ассетов и режим выполнения.

## Данные и аналитика

В `SQLite` создаются и поддерживаются таблицы:

- `route_strategies` - базовая стратегия для маршрута;
- `metrics` - серверные и клиентские performance-метрики;
- `page_views` - просмотры страниц;
- `product_events` - пользовательские события внутри сценариев.

Админка поддерживает:

- переключение базовой стратегии маршрута;
- сброс накопленных метрик и событий;
- просмотр последних записей;
- экспорт сырых метрик в `CSV`;
- экспорт агрегированной сводки и воронки.

## A/B-эксперименты

Для каждого сценария в `server/src/experiments/config.ts` описан активный эксперимент, который распределяет пользователя по группам:

- `group_csr -> csr`
- `group_ssr -> ssr`
- `group_ssg -> ssg`

Назначение сохраняется в cookie, поэтому один и тот же пользователь получает стабильную стратегию между заходами.

## SSG

Статическая генерация выполняется скриптом `server/src/strategy/adapters/ssg.ts`. Во время сборки проект заранее рендерит HTML для всех известных сценариев и кладет результат в `server/static/page/`.

Если для стратегии `ssg` статический файл найден, сервер отдает его и подставляет актуальный runtime-мета-блок для метрик и эксперимента. Если файла нет, сервер делает fallback на `SSR`.

## API и служебные маршруты

- `GET /api/page/:slug` - данные сценария;
- `POST /api/metrics` - прием performance-метрик;
- `POST /api/events` - прием продуктовых событий;
- `GET /admin` - HTML-админка;
- `GET /admin/summary.json` - агрегированная performance-сводка;
- `GET /admin/product-summary.json` - сводка по продуктовым событиям;
- `GET /admin/funnel.json` - воронка сценария;
- `GET /healthz` - healthcheck.
